# EAS preview builds — TestFlight & Play Internal (P-4.6)

This checklist supports **`eas build --profile preview`** from the architecture roadmap. Human steps (Apple Developer, Google Play Console) cannot be automated in-repo; complete them once per organization.

## Prerequisites

1. **Expo account** and **EAS** enabled for the project (`npx eas-cli login`).
2. Link the app: from `mobile/`, run `npx eas-cli init` if the project has no `extra.eas.projectId` yet (Expo will merge project id into `app.json` / `app.config`).
3. **Apple Developer Program** membership (paid) for TestFlight distribution.
4. **Google Play Console** account with an app created for package **`lu.verve.app`** (must match `app.json`).

## Environment secrets (EAS)

Configure in the Expo dashboard or via CLI (do not commit tokens):

| Secret | Use |
|--------|-----|
| `EXPO_PUBLIC_SUPABASE_URL` | Production / preview Supabase URL |
| `EXPO_PUBLIC_SUPABASE_ANON_KEY` | Anon key for the same project |
| `EXPO_PUBLIC_SENTRY_DSN` | Optional; error monitoring (P-4.4) |
| `SENTRY_AUTH_TOKEN` | Optional; upload source maps on native build (Sentry plugin) |

```bash
cd mobile
npx eas-cli secret:create --name EXPO_PUBLIC_SUPABASE_URL --value "https://....supabase.co" --type string
# repeat for other secrets
```

## Build commands (from `mobile/`)

Internal **preview** binaries (roadmap P-4.6):

```bash
npx eas-cli build --profile preview --platform ios
npx eas-cli build --profile preview --platform android
```

Or both:

```bash
npx eas-cli build --profile preview --platform all
```

- **iOS:** EAS produces an `.ipa` suitable for **TestFlight** internal testing after you complete App Store Connect setup below.
- **Android:** `eas.json` uses **`app-bundle`** for the `preview` profile so the roadmap command produces an AAB suitable for Play Internal Testing. Use the separate `internal-apk` profile only for quick sideload/device-smoke builds.

## iOS — TestFlight (internal)

1. In **App Store Connect**, create an app with bundle id **`lu.verve.app`** (matches `app.json`).
2. Accept agreements, fill privacy labels, and add **Internal Testing** group testers (Apple IDs).
3. After the first successful **`preview`** iOS build, submit to TestFlight:

   ```bash
   npx eas-cli submit --platform ios --latest
   ```

   Or attach the build manually in App Store Connect → TestFlight.

4. **PASS** pilot gate when internal testers can install from TestFlight and complete `mobile/docs/PILOT_TEST_PLAN.md` on a physical device.

## Android — Play Console internal testing

1. Create an application with package **`lu.verve.app`**.
2. Complete the Play Console **dashboard** checklist (store listing can be minimal for internal track).
3. Create an **Internal testing** release; upload the **AAB** generated by `npx eas-cli build --profile preview --platform android`.
4. For sideload-only smoke tests, run `npx eas-cli build --profile internal-apk --platform android`; do not upload that APK to Play Internal Testing.
5. Add tester emails to the internal track; distribute the opt-in link.

## Profiles reference (`eas.json`)

| Profile | Purpose |
|---------|---------|
| `development` | Dev client install for on-device debugging with native modules |
| `preview` | **P-4.6** — internal distribution, roadmap `preview` |
| `internal-apk` | Android sideload smoke builds only; not for Play Internal Testing |
| `production` | Store release track; version auto-increment |

## After builds

- Run **`mobile/docs/PILOT_TEST_PLAN.md`** again on **store-equivalent** builds (not only Expo Go) before widening the tester pool.
- File `FAIL` rows in Notion with owner + ETA per sign-off rule in the pilot plan.
